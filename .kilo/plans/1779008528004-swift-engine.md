# Plan: Merge Landing Pages into Config-Driven Architecture

## Analysis Summary

### Two existing landings:
1. **Biomedical** (`/biomedical`) — SST Men, medical-focused, custom CSS classes, Fancybox, no Framer Motion
2. **Life-style** (`/life-style`) — SST 90-day challenge, Framer Motion, react-icons, shadcn/ui, more deps

### Key findings:
- Both use **TanStack Router + Vite + Tailwind CSS v4 + React 19**
- **Sections are fundamentally different** between landings (different JSX, different styling, different content) — they cannot be merged into shared components
- Shared infrastructure is needed: routing, theming, config, common UI
- Biomedical uses custom CSS component classes (`.btn-cta`, `.section-label`); Lifestyle uses Tailwind utilities + Framer Motion
- Data: Biomedical is hardcoded in components; Lifestyle uses separate data files

### Section mapping (by function, NOT shared code):

|----------|-----------|-----------|
| Nav | `Navbar.tsx` | `Navbar.jsx` |
| Hero | `Hero.tsx` (video + reviews) | `Hero.jsx` (2-col orange+image) |

---

## Architecture Decision

**Strategy**: Create a new merged project (`/merged`) in the workspace root. Use life-style as the base (it has more dependencies) and layer in biomedical's unique deps.

**Why not share sections?** The sections have completely different JSX structures, styling, and content. Forcing them into "unified" components would break existing UIs. Instead, each landing keeps its own section implementations, organized by slug.

## User Decisions

1. **Project location**: New `/merged` folder at workspace root
2. **API strategy**: Scaffold only — create `createClient()` factory and hooks ready for future use; keep using static data files
3. **CSS strategy**: Convert biomedical's custom CSS classes (`.btn-cta`, `.section-label`, `.h2-display`, `.container-narrow`) to Tailwind utility classes in each biomedical component, matching equivalent visual output

2. **Merge `package.json`** — combine all dependencies from both:

4. **Create `tsconfig.json`** — merge compiler options

### Phase 6: LandingRenderer

**`/src/LandingRenderer.jsx`**:
```
- Read slug from route params
- Look up config from landings.js
- If not found → 404
- Apply theme via ThemeProvider
- Lazy-load the landing's SECTION_MAP
- Render sections dynamically based on config.sections
- Wrap each section in Suspense with SectionSkeleton
- Render layout components (Footer, WhatsApp) if config says so
- Set document meta from config
```

### Phase 7: Routing

**`/src/routes/$slug.tsx`**:
- TanStack Router file-based route with `$slug` param

**`/src/routes/__root.tsx`**:

**`/src/routes/index.tsx`** (optional):

```js
export function createClient(apiKey) {
  const baseURL = "https://api.example.com";
  return {
    async get(endpoint) {
      const res = await fetch(`${baseURL}${endpoint}`, {
        headers: { "Authorization": `Bearer ${apiKey}` },
      });
      return res.json();
    },
  };
}
```

Currently both landings use static data. The API layer is scaffolding for when dynamic data is needed. It can be used in a `useLandingData` hook.

## Open Questions (Resolved by Analysis)

- **Will sections be shared?** No — each landing has unique section implementations. The SECTION_MAP pattern allows using the same key name (e.g., "guarantee") mapping to different components per landing.
- **Will there be CSS conflicts?** Minimal. Biomedical's custom CSS classes are converted to Tailwind utilities. Theme variables are applied dynamically per-route.
- **What about the `container-narrow` class?** Converted to Tailwind utilities inline in biomedical components: `mx-auto w-full max-w-[480px] md:max-w-[720px] lg:max-w-[960px] xl:max-w-[1140px]`.

## Risk Mitigation

- **UI preservation**: Each section is copied as-is; only imports and CSS classes change
- **CSS class conversion**: Biomedical's custom classes are carefully mapped to equivalent Tailwind utilities preserving exact visual output
- **Theme isolation**: CSS variables are applied per-route via ThemeProvider
- **Lazy loading**: All sections use `React.lazy` + `Suspense` for performance
- **Adding new landings**: Just add a new entry in `landings.js`, create a new folder under `/src/landings/`, and define a SECTION_MAP
- **Custom fonts**: Biomedical's `@font-face` declarations stay in `styles.css`; the `.regular`/`.medium`/`.bold` utility classes remain as they map to custom font families
